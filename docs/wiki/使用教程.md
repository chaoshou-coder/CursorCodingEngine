# 使用教程

本教程通过完整示例，手把手教你从安装到完成一次开发任务的全流程。

---

## 一、环境准备

### 1.1 安装

```bash
# 克隆仓库
git clone https://github.com/chaoshou-coder/CursorCodingEngine.git
cd CursorCodingEngine

# 创建虚拟环境（建议命名 CCEfor<项目名>）
python -m venv CCEforCursorCodingEngine

# 激活（Windows PowerShell）
.\CCEforCursorCodingEngine\Scripts\Activate.ps1
# 激活（Linux/macOS）
# source CCEforCursorCodingEngine/bin/activate

# 安装
pip install -e .
```

### 1.2 验证

```bash
codingengine --help
# 应看到 init、emit、run、status、tail、list、stats 等子命令
pip show codingengine
# 确认 Location 指向 CursorCodingEngine
```

## 二、完整示例：从需求到完成

假设你要实现「为项目添加日志配置模块」。

### 2.1 初始化 run

```bash
# 在项目根目录执行，用简短英文描述
codingengine init "Add logging config module"
```

**输出示例**：
```
run_id: 20250305_143022_a1b2c3d4
```

**重要**：记下 `run_id`，后续所有 `emit` 命令都要带 `--run-id <run_id>`。

### 2.2 plan 阶段：制定计划

**步骤 1**：开始 plan 阶段

```bash
codingengine emit stage.started --stage plan --run-id 20250305_143022_a1b2c3d4
```

**步骤 2**：分析需求，拆分任务

按 `context_limit` 将需求拆成可独立完成的小任务。例如：

| task_id | 描述 | 依赖 |
|---------|------|------|
| task_001 | 创建 logging 配置类 | 无 |
| task_002 | 在 main 中集成 logging | task_001 |

**步骤 3**：编写 plan.json

在 `~/.codingengine/runs/20250305_143022_a1b2c3d4/artifacts/` 下创建 `plan.json`：

```json
{
  "tasks": [
    {
      "id": "task_001",
      "description": "Create logging config class in src/utils/logging_config.py",
      "estimated_context": 4000,
      "files": ["src/utils/logging_config.py"]
    },
    {
      "id": "task_002",
      "description": "Integrate logging in main entry point",
      "estimated_context": 3000,
      "files": ["src/main.py"]
    }
  ],
  "dependencies": {
    "task_002": ["task_001"]
  },
  "atomic_commit": true,
  "rollback_strategy": "git_revert"
}
```

**步骤 4**：完成 plan 阶段

```bash
codingengine emit stage.completed --stage plan --run-id 20250305_143022_a1b2c3d4
```

### 2.3 develop 阶段：按任务开发

**步骤 1**：开始 develop 阶段

```bash
codingengine emit stage.started --stage develop --run-id 20250305_143022_a1b2c3d4
```

**步骤 2**：执行 task_001

1. 创建 `src/utils/logging_config.py`，实现配置类
2. **立即 lint**：`ruff check src/utils/logging_config.py`
3. 调用 **code-simplifier**（若在 Cursor 中，@code-simplifier 清理该文件）
4. **原子提交**：
   ```bash
   git add -A
   git commit -m "[codingengine] task_001"
   ```

**步骤 3**：执行 task_002

1. 修改 `src/main.py`，集成 logging
2. 立即 lint
3. code-simplifier 清理
4. 原子提交：`git commit -m "[codingengine] task_002"`

**步骤 4**：完成 develop 阶段

```bash
codingengine emit stage.completed --stage develop --run-id 20250305_143022_a1b2c3d4
```

### 2.4 verify 阶段：验证

**步骤 1**：开始 verify 阶段

```bash
codingengine emit stage.started --stage verify --run-id 20250305_143022_a1b2c3d4
```

**步骤 2**：运行全量 lint 和测试

```bash
ruff check .
pytest
```

**步骤 3**：若通过，完成 verify

```bash
codingengine emit stage.completed --stage verify --run-id 20250305_143022_a1b2c3d4
```

**步骤 4**：若失败

- 诊断失败原因
- `git revert` 回滚失败任务对应的 commit
- 修复后重试（最多 3 轮）

### 2.5 完成

```bash
codingengine emit run.completed --run-id 20250305_143022_a1b2c3d4
```

## 三、在 Cursor 中使用

### 3.1 启用 Skill

CursorCodingEngine 已包含 `.cursor/skills/codingengine/` 和 `code-simplifier/`。在 Cursor 中打开本项目时，Agent 会自动读取这些 Skill。

### 3.2 触发方式

在 Cursor Chat 中任选其一：

- 说「按计划执行」「开发计划现在执行」
- @ `.cursor/plans/xxx.plan.md`（若已有计划文档）
- 直接描述需求，如「添加用户认证模块」

### 3.3 Agent 会做什么

Agent 按 codingengine Skill 执行：

1. 调用 `codingengine init "简短描述"`
2. 取得 run_id 后，按阶段 emit
3. plan：分析需求，产出 plan.json
4. develop：按任务修改 → lint → code-simplifier → 原子提交
5. verify：全量测试，失败则诊断回滚重试
6. 完成：`codingengine emit run.completed`

### 3.4 在其它项目中使用

若要在**其它项目**中使用 CodingEngine：

1. 在目标项目根目录激活 venv，安装 codingengine：`pip install -e /path/to/CursorCodingEngine`
2. 将 `CursorCodingEngine/.cursor/skills/` 复制到目标项目的 `.cursor/skills/`
3. 在 Cursor 中打开目标项目，按上述方式触发

## 四、使用 codingengine run 一键执行

若不想手动 emit，可用 `codingengine run` 自动执行阶段机：

```bash
codingengine init "Add logging config"
# 取得 run_id

codingengine run "Add logging config" --resume
# 或指定 run_id 后，run 会按 plan → develop → verify 自动执行
```

**注意**：`run` 依赖阶段处理器实现，当前可能为 stub。手动 emit 更可控。

## 五、断点续传

若执行到一半中断（如 develop 未完成）：

```bash
# 查看状态
codingengine status

# 查看事件流
codingengine tail --follow

# 从上次中断处继续
codingengine run "需求" --resume

# 或从指定阶段重新开始
codingengine run "需求" --from-stage develop
```

## 七、plan.json 字段说明

| 字段 | 必填 | 说明 |
|------|------|------|
| tasks | 是 | 任务列表，每项含 id、description、estimated_context、files |
| dependencies | 否 | 任务依赖，如 `{"task_002": ["task_001"]}` |
| atomic_commit | 是 | 建议 true，每任务独立 commit |
| rollback_strategy | 是 | 建议 `"git_revert"`，失败时回滚 |
| decoupled | 否 | 模块解耦信息，减少跨模块修改 |

## 八、常见问题

### init 后找不到 run_id

输出中会显示 `run_id: xxxxx`，复制整行或仅 ID 即可。

### emit 报错「run_id 不存在」

确认 run_id 正确，或先 `codingengine list` 查看已有 run。

### 产物目录在哪

`~/.codingengine/runs/<run_id>/artifacts/`。Windows 下为 `C:\Users\<用户名>\.codingengine\runs\`。

### 如何修改数据目录

设置环境变量 `CODINGENGINE_DATA`，如 `export CODINGENGINE_DATA=/data/codingengine`。
